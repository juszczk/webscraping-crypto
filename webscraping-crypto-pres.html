Web Scraping and Social Media Scraping¶

Final Project: A Case of Difficulty Web Structures and Limitations¶

by Marina Juszczyk (412412)

Throughout the course, many techniques and issues of handling basic web crawling and scraping were presented. There are websites that gathers interesting and useful data – like in the following project: prices of cryptocurrencies – yet, to extract information is much harder and challenging than it initially appears. Following project focuses on a website CoinMarketCap (https://coinmarketcap.com/), one of the most influential ranking webs on the crypto market. It stores real-time and historical data regarding prices, volume and market capitalization for over two hundred existing currencies.

It is not, however, the main target of this work to gather prices of crypto currencies. In fact, it can be easily done by numerous, easily accessible APIs. Main challenges and goals were to enter a website with HTTPS protocol and many Java Script elements, and to bypass all the possible issues. Those two criteria are not random: HTTPS protocol tends to slow down the website loading time (obtaining a security certificate), and advanced Java Script site masks many elements and has XPaths re-generated dynamically. Apart from advanced user authentication, it is hard to imagine more challenging object for data crawling. As expected, mentioned webiste comply with those expectations.

The code crawls through the website, access subpages for chosen currencies (set by the user) and parse following information into a data frame: name, ticer, opening price, closing price, hight price, low price, volume and market capitalization. To test the gathered data, simple computations between variables, and plots are performed and presented.

In this presentation, final code is presented in a form of snipptes to explain its most crucial elements. They might not appear in chronological order, so in case of confusion, please treat the source code as a benchmark. Full code (without comments) is available separately. Following project uses Selenium with Chrome WebDriver, as it is the most adequate tool to scrape this type of website.

Bottleneck 1: Dynamic Xpaths¶

Unfortunately, Selenium does not handle generating dynamic Xpaths well. Especially, if the dynamic element is inside the path. Although, it is not the best practice to use the whole path, for this particular webiste any other solutions were resulting in a crash of the code. For this reason, many Xpaths were genrated manually by adding parts of strings, with dynamic element included.

Bottleneck 2: Page Loading¶

For this particular website, there is no problem with pagination as there are no pages - if so, in a form of tabs with assigned Xpaths. CoinMarketCap creates another obstacle, JS elements are not loading or are masked/hidden, until certain pixel is reached. For that reason, code needs to mimic human-like behaviour and scroll through the page. One important note that must be included, is that due to obtaining a security certificate all the possible activities are blocked. Therefore, there must be a break between actions to not crash the code. As the photo below presents, the machine is not able to access the Historical Data button, untill the graph from the currency webpage is fully loaded - in many runs of the code, the graph was not loading until page was scrolled down to the certain point (the optimal pixel point is floating over time).

Bottleneck 3: Masked and Hidden Elements¶

The biggest challenge in this project is handling masked elements. Due to the structure and dinamic character of the website, certain buttons and tables are not accessible until other element is not loaded (usually on a given position, e.g. Bottleneck 1). Simply, the machine is not able to access chosen element without pre-checking that related elements are not visible. For this project, it appears that time.sleep() method was more effective. It is suspected, that WebDriverWait.until() is slower, as the relative element might be depended on the third element, that was not recognized in the code.

It turned out, that webiste only responds to another human-like behaviour, and that sendig an empty key (mimics "Enter" button). Another one, caret for period selector is initally masked and must be approached with an actual mouse movement.

Bottleneck 4: Cookies¶

Luckly, cookies on this website were not troublesome iteself. For the neatness reasons, after the session all cookies are erased. However, the pop-up window regarding cookies were slowing down the crawling process massively. If not clicked and closed, there were multiple runs when the code broke down as the website slowed down the loading.
